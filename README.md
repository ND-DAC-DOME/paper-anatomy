# Paper Anatomy Vocabulary (PAX)

A FAIR, ontology-grounded vocabulary and JSON-LD schema for describing the **structural and rhetorical anatomy of a scientific paper** — sections and their hierarchy, and every element inside each section (paragraphs, figures, tables, captions, formulas, footnotes…), anchored to page images with bounding boxes and full processing provenance.

- **Namespace**: `https://w3id.org/paper-anatomy/vocab#` (prefix `pax:`)
- **Shapes namespace**: `https://w3id.org/paper-anatomy/shapes#` (prefix `paxs:`)
- **Status**: v0.2.0 — reviewer-approved **testing** vocabulary; w3id registration deliberately deferred (see [w3id note](#fair-compliance))

PAX itself is small by design: it only defines what established community vocabularies cannot express. Everything else is typed with SPAR (DoCO, DEO, FaBiO, PO) and W3C (Web Annotation, PROV, SKOS) terms.

Two purposes:

1. **Interchange format** — a pipeline-independent description of "what is in this paper and where", usable by any downstream system (viewer, biocuration, RAG).
2. **Evaluation ground truth** — a new AI pipeline or an LLM can be scored by generating this same graph shape and diffing it against a curated reference (see [Evaluation](#using-the-graph-to-evaluate-ai-pipelines)).

## Producers

The vocabulary is application-independent. Instance data is minted by applications under **their own** namespace (the exporter default is `https://w3id.org/alexandria/paper/…`, the namespace of the Alexandria application; any deployment can override it with `--base-iri`).

Current producers:

- **deepseek-ocr-experiments** — `kg/export_kg.py` converts its `processed_document.json` into this schema; that repo also hosts the interactive explorer and the PPTX deck generator. The [example graph](examples/Mercadier2011.jsonld) here was generated by it.
- **paper-atomizer** (crcresearch) — [`adapters/paper_atomizer.py`](adapters/paper_atomizer.py) converts its `processed_document.json` (flat elements, `{x,y,width,height}` boxes, string IDs, nested section tree) into the same graph. **This is the realized proof that the KG is the pipeline-independent layer**: an independently evolved pipeline reaches a graph with 0 SHACL violations and 0 warnings, diffable against any other producer's. Graphs are built through the LinkML-generated Pydantic models, so malformed output fails at construction.
- **Alexandria** (planned) — the production application; same adapter pattern.

## Why these ontologies (and not others)

Requirement: community-adopted, actively maintained vocabularies — no dead one-off schemas.

| Vocabulary | Prefix | What we use it for | Adoption / liveness evidence |
|---|---|---|---|
| **DoCO** — Document Components Ontology | `doco:` | Structural element types: `Section`, `Paragraph`, `Figure`, `Table`, `Formula`, `Footnote`, `List`, `Title`, `SectionTitle`, `FrontMatter/BodyMatter/BackMatter`, `Abstract`, `Bibliography`, `Label`, `FigureBox`, `TableBox` | Part of the SPAR suite (OpenCitations / U. Bologna). **v1.4.0, last modified 2026-06-25** — actively maintained. Used by Europe PMC, OpenCitations, GROBID-derived tooling. |
| **DEO** — Discourse Elements Ontology | `deo:` | Rhetorical typing of sections (`Introduction`, `Methods`, `Results`, `Discussion`, `Conclusion`, `Acknowledgements`, …) and `deo:Caption` | SPAR suite; imported by DoCO itself. 39 discourse classes. |
| **FaBiO** — FRBR-aligned Bibliographic Ontology | `fabio:` | Document typing (`ResearchPaper`, `JournalArticle`), `fabio:Page`, `fabio:hasPageCount` | SPAR suite; the de-facto RDF vocabulary for scholarly artifacts. |
| **Pattern Ontology** | `po:` | `po:contains` — the containment relation DoCO itself is built on (paper ⊃ matter ⊃ section ⊃ element) | Dependency of DoCO; same maintainers. |
| **W3C Web Annotation** | `oa:` | Anchoring elements to page-image regions: `oa:ResourceSelection` + `oa:FragmentSelector` | W3C Recommendation (2017). The backbone of IIIF, Hypothes.is, Europeana. |
| **W3C Media Fragments** | — | `xywh=x,y,w,h` selector syntax inside the FragmentSelector | W3C Recommendation; universally used in IIIF. |
| **W3C PROV-O** | `prov:` | Which pipeline (commit) and which models produced each artifact | W3C Recommendation; the standard for provenance. |
| **DCMI Terms** | `dcterms:` | `title`, `description`, `identifier`, `license`, `source` | Universal. |
| **Schema.org** | `schema:` | `schema:position` (reading order), `schema:image` (page renders), interop with search engines | Universal. |

**Rejected alternatives**: SALT (abandoned), OntoDM (wrong domain), custom-only schema (fails FAIR-I). TEI (GROBID's format) is excellent but XML-native and not RDF; DoCO actually provides mappings from TEI concepts.

Everything the standard vocabularies cannot express lives in one clearly-marked extension namespace:

- `pax:` — *Paper Anatomy Vocabulary* (`https://w3id.org/paper-anatomy/vocab#`, w3id registration pending). Covers: `pax:figureType` (SKOS scheme), `pax:chartData`, `pax:printedPageLabel`, `pax:region`, `pax:pageImageWidth/Height`, `pax:PageHeader`, `pax:PageFooter`, `pax:CodeBlock`, and the OCR `pax:rawType` passthrough. Formally defined in [`pax.ttl`](./pax.ttl).

## Graph shape

```
schema:Dataset + prov:Entity  (…/run/<run-id> — the KG the pipeline generated)
        │ prov:wasGeneratedBy ─▶ prov:Activity (commit, models) ─▶ prov:SoftwareAgent ×5
        │ dcterms:license CC-BY-4.0        (license of the EXTRACTED GRAPH, not the paper)
        └ schema:about ─▶ fabio:JournalArticle (the article EXPRESSION)

fabio:JournalArticle ──dcterms:title──▶ "Origin of uranium deposits…"
        │ frbr:embodiment ─▶ fabio:DigitalManifestation (the PDF) ──frbr:part──▶ pages
        │ fabio:hasPageCount 6
        │ po:contains
        ├─▶ doco:FrontMatter ──po:contains──▶ doco:Section "ABSTRACT" (also doco:Abstract)
        ├─▶ doco:BodyMatter  ──po:contains──▶ doco:Section "Introduction" (also deo:Introduction)
        │                                     doco:Section "Results" (also deo:Results)
        │                                        │ po:contains
        │                                        ├─▶ doco:Section "Controls on REE…"  (subsection)
        │                                        ├─▶ doco:Paragraph …
        │                                        └─▶ doco:FigureBox
        │                                              ├─▶ doco:Figure  (pax:figureType pax:lineGraph,
        │                                              │                 pax:chartData {…},
        │                                              │                 pax:region → oa:ResourceSelection
        │                                              │                    └ oa:hasSelector xywh=111,128,972,83
        │                                              │                    └ oa:hasSource  page_003.png)
        │                                              └─▶ deo:Caption "Fig. 3 Chondrite-normalised…"
        └─▶ doco:BackMatter  ──po:contains──▶ doco:Section "Supporting Information"

fabio:Page (×6)  schema:position 1…6 · pax:printedPageLabel "264"…"269" · pageImageWidth/Height · schema:image
```

Key modelling decisions:

1. **Containment via `po:contains`** — the property DoCO is defined around. The chain is `paper → matter → section → (subsection | element)`. Elements not belonging to any section attach directly to their matter or the paper.
2. **Captions use the DoCO box pattern**: a `doco:FigureBox` contains the `doco:Figure` *and* its `deo:Caption` (same for `doco:TableBox`). This preserves caption linking in the standard way.
3. **Rhetorical double-typing**: a section named "Introduction" is typed both `doco:Section` (structure) and `deo:Introduction` (discourse role). Name→DEO mapping is heuristic and optional.
4. **Bounding boxes reuse W3C Web Annotation components as a compact application profile**: each element carries `pax:region` → `oa:ResourceSelection` (source = page image, selector = `oa:FragmentSelector` with Media Fragments `xywh=x,y,w,h`). This is deliberately **not** a conforming `oa:Annotation` — generic annotation clients and IIIF viewers will not consume it as-is; a lossless transform to conforming Annotations/AnnotationPages is straightforward if that integration is ever needed. Coordinates live in the pixel space declared by `pax:pageImageWidth/Height` on the page.
5. **Reading order = `schema:position`** (global integer per element).
6. **Two page numberings, explicitly separated**: `schema:position` (sequence 1..N) vs `pax:printedPageLabel` (the label as printed — possibly non-numeric: "xiii", "S12", "266a"). `pax:printedPageNumber` is a numeric identifier parsed from the label when available (e.g. the 1 in "1 / 21"); the label remains authoritative.
7. **Provenance sits on the KG dataset node, not the article**: the pipeline generated the *extracted graph*, not the scholarly work. A `schema:Dataset`/`prov:Entity` node carries `prov:wasGeneratedBy` (activity: pipeline commit + every model version) and the dataset license, and is `schema:about` the article. Generated nodes are **run-scoped** (`…/run/<run-id>/…`) because element segmentation is not stable across reruns; article/manifestation/SI IRIs stay stable.
8. **`chart_data` embedded as a JSON literal** (`"@type": "@json"`, JSON-LD 1.1) — extracted chart points are payload, not graph structure. Consumers that care can parse; graph tools can ignore.

### Named design patterns (ODPs)

The decisions above are instances of recurring, checkable patterns — named here so a reader
(or an agent) can recognize them wherever they recur, in the spirit of Modular Ontology
Modeling. Formal OPLa annotations on the vocabulary itself are a v0.3 candidate.

| Pattern | Instantiated as | Source / precedent |
|---|---|---|
| **Box grouping** | `doco:FigureBox`/`doco:TableBox` containing the figure/table *and* its caption (`po:contains` + `dcterms:hasPart`) | published DoCO example |
| **Rhetorical double-typing** | `doco:Section` + `deo:Introduction`/`deo:Results`/… on the same node | canonical SPAR pattern |
| **Compact application profile** | `pax:region` → `oa:ResourceSelection` + `oa:FragmentSelector` (Media Fragments `xywh`), deliberately **not** a conforming `oa:Annotation` | W3C Web Annotation, profiled |
| **FRBR strata** | Expression root (`fabio:JournalArticle`) → `frbr:embodiment` → PDF `fabio:DigitalManifestation` → pages as `frbr:part`; Work node optional | FaBiO / FRBR |
| **Dataset-scoped provenance** | a `schema:Dataset`+`prov:Entity` node carries `prov:wasGeneratedBy` and the license; activities carry `prov:used`; the article stays clean | PROV-O; separates the extracted graph from the scholarly work |
| **Run-scoped minting** | regenerable nodes under `…/run/<id>/…`; work-level IRIs (paper, manifestation, SI) stay stable | segmentation is unstable across reruns |
| **SKOS controlled list** | `pax:FigureTypeScheme` (14 concepts; `skos:notation` preserves pipeline codes) | SKOS best practice for closed value sets |
| **Materialized projections** | `pax:level`/`matter`/`pageStart`/`pageEnd` as flat classifier outputs, consistency-guarded by the recommended SHACL profile | evaluation targets, not reasoning shortcuts |
| **Layered supplements** | the SI *description section* ≠ the SI *artifact* (`fabio:SupplementaryInformationFile` + `frbr:supplementOf`); processed SI recurses into a full graph | FRBR + DataCite `IsSupplementTo` practice |

| Section `level` | RDF type |
|---|---|
| `title` | (section node kept as `doco:Section`; its name feeds `dcterms:title`) |
| `label` | `doco:Label` |

Name → DEO heuristics (case-insensitive, applied when confident): `abstract → doco:Abstract`, `introduction → deo:Introduction`, `materials → deo:Materials`, `methods / materials and methods → deo:Methods`, `results → deo:Results`, `discussion → deo:Discussion`, `conclusion(s) → deo:Conclusion`, `acknowledg… → deo:Acknowledgements`, `related work → deo:RelatedWork`, `background → deo:Background`, `references / bibliography → doco:Bibliography`, `supporting information / supplementary… → deo:SupplementaryInformationDescription`.

## Supplementary material

Scientific SI is where the data payload often lives — in Mercadier 2011, the main text references it **16 times** (`Table S1` ×3, `Table S2` ×2, `Figure S1` ×2, "Supplementary Material" ×7, "Supporting Information" ×2), and the 66 REE analyses central to the paper are in Table S1/S2, not in the body. The KG models it in layers, keeping one crucial distinction: the *section describing the SI* inside the main text is *not* the SI itself — the SI is a **separate digital object** with its own identity and provenance.

| Layer | What | Vocabulary | Status |
|---|---|---|---|
| 0 | The describing section in the main text | `deo:SupplementaryInformationDescription` (name heuristic) | ✅ implemented |
| 1 | Stub node for the SI artifact even when unprocessed: title, DOI/URL, link to the paper | `fabio:SupplementaryInformationFile` + `frbr:supplementOf` (inverse `frbr:supplement` on the paper) — the FRBR relation matching DataCite's `IsSupplementTo` practice | ✅ implemented in the exporter (`--supplement-label/-doi/-url`) |
| 2 | SI processed by a pipeline → **full recursive document graph** in the same schema (own matters, sections, elements, pages, prov activity) under `…/supplement/1/` | none new — the schema recurses | ✅ implemented in the exporter (`--supplement-dir`) |
| 3 | Resolved cross-references: the in-text mention "Table S1" → the actual `doco:Table` node inside the SI graph | `dcterms:references` | future — and a natural evaluation metric ("can the pipeline link mentions to targets?") |

The [`examples/Mercadier2011.jsonld`](examples/Mercadier2011.jsonld) carries a layer-1 stub (the SI file itself has not been run through a pipeline yet).

## FAIR compliance

| Principle | How |
|---|---|
| **F**indable | Every node has a stable IRI: vocabulary terms under `https://w3id.org/paper-anatomy/…`, instance data under the producing application's base (default `https://w3id.org/alexandria/paper/<id>/…`); the paper node carries `dcterms:identifier` (DOI when known). |
| **A**ccessible | Plain JSON-LD file over HTTP; no proprietary format. |
| **I**nteroperable | All structure expressed in W3C/SPAR vocabularies; one documented extension namespace; JSON-LD 1.1 → trivially convertible to Turtle/N-Quads for any triple store. |
| **R**eusable | `dcterms:license` on the dataset node; full PROV provenance (pipeline commit + model versions) so any consumer knows exactly how the graph was produced. |

> **w3id note**: `https://w3id.org/paper-anatomy/` (vocabulary) and `https://w3id.org/alexandria/` (instance data) are the permanent-identifier bases. Registering them is a one-time PR to [w3id.org](https://github.com/perma-id/w3id.org) — draft `.htaccess` files live in [`w3id/`](w3id/). **Deliberately not registered yet** — pending GitHub Pages, target-URL confirmation, and a one-time OOPS! design-review run.

## Using the graph to evaluate AI pipelines

The graph gives you typed nodes + typed edges + spatial anchors, which decomposes "did the model understand the document structure?" into measurable sub-tasks:

| Sub-task | Metric over the graph |
|---|---|
| Element detection | Precision/recall of `(type, page)` node pairs vs reference |
| Localization | IoU between `xywh` selectors of matched elements |
| Reading order | Kendall-τ between `schema:position` sequences |
| Section hierarchy | Tree edit distance between the `po:contains` section trees |
| Rhetorical labelling | Accuracy of `deo:*` / matter types on matched sections |
| Caption linking | Precision/recall of FigureBox/TableBox membership edges |
| Chart extraction | Downstream comparison of `pax:chartData` payloads (point-count, value error) |

Workflow: run candidate pipeline → export to this schema (via the deepseek-ocr-experiments exporter or an adapter for the pipeline's native schema) → graph-diff against the curated reference JSON-LD of the same paper. Since both sides use the same context, the comparison is schema-free:

```bash
python3 eval/evaluate.py reference.jsonld candidate.jsonld -o report.json [--iou 0.5]
```

Elements are matched greedily within (page, type) groups by descending IoU; detection counts matches at the IoU threshold, and the remaining metrics are computed over those matches (localization, reading order) or over titles/boxes/payloads (hierarchy, rhetorical, caption links, chart data). Every metric was verified against a targeted perturbation of the real example (`eval/test_evaluate.py`).

## Formal-check findings worth knowing

- **The PAX ontology document is OWL 2 DL. The combined external-vocabulary closure is not OWL 2 DL** (~1.1k violations inherited from legacy imports — foaf and dcterms model their terms as plain `rdf:Property`). The claim is deliberately scoped to the document: PAX has no `owl:imports`, and the external declarations in `pax.ttl` act as local OWL-category compatibility declarations.
- **HermiT finds the *transformed* pinned closure consistent with no unsatisfiable classes** — "transformed" because 798 Pattern-Ontology SWRL triples are removed before reasoning (built-in atoms unsupported by HermiT-in-ROBOT).
- `pax:chartData` intentionally has **no range axiom**: `rdf:JSON` sits outside the OWL 2 datatype map and a range would break the DL profile. The JSON-literal intent is documented on the property; the `rdf:JSON` datatype is enforced at the instance level by `paxs:ChartDataShape` (SHACL), which is negative-tested in `validate.py`.
- External terms used by pax.ttl carry explicit category declarations (AnnotationProperty/Class/…) — without them OWLAPI guesses and reports punning.

## Known limitations / next steps

- The DEO mapping is heuristic; a curator should be able to override (that is the point of the reference graphs being editable).
- The scoring engine (`eval/evaluate.py`) covers the seven metrics; so far it has only been exercised against perturbations of one reference graph — the paper-atomizer adapter will provide the first real cross-pipeline comparison.
- Supplementary-material layer 3 (mention → target cross-references) not implemented; see [Supplementary material](#supplementary-material).
- Publication path: make the repo public → enable GitHub Pages (from `/docs`) → w3id PR (`w3id/` drafts). The Widoco site and the archived OOPS! review (`docs/resources/quality/0.2.0/oops-disposition.md` — 0 critical, all findings dispositioned) are done.
- v0.3 candidates: mutual `owl:disjointWith` among the five PAX element classes (OOPS P10, backward-compatible); formal OPLa annotations for the named design patterns; LinkML on the evaluation-engine side (consumer models), not for vocabulary authoring.
