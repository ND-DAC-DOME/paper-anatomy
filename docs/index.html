Paper Anatomy Vocabulary (PAX)

Testing release v0.2.0 — not yet formally published

A FAIR, ontology-grounded vocabulary and JSON-LD schema for the structural anatomy of a scientific paper — section hierarchy, every element inside each section, captions, page-image regions, and full processing provenance. PAX defines only what established community vocabularies cannot express; everything else is typed directly with SPAR and W3C terms.

  • 📄 Ontology specification (Widoco-generated; serializations: Turtle · JSON-LD · RDF/XML · N-Triples)
  • 🔎 SHACL validation profiles (core · recommended · diagnostic)
  • 🗄️ Version snapshot 0.2.0 (the owl:versionIRI target)
  • 💻 Source repository

Status and conformance

  • Canonical namespace: https://w3id.org/paper-anatomy/vocab#, preferred prefix pax. Shapes namespace: https://w3id.org/paper-anatomy/shapes#, prefix paxs. (w3id registration pending — deferred until this site is publicly served.)
  • The PAX ontology document is OWL 2 DL. The combined external-vocabulary closure is not OWL 2 DL (≈1.1k violations inherited from legacy imports such as foaf and dcterms). The claim is deliberately scoped to the document: PAX has no owl:imports, and the external declarations in pax.ttl act as local OWL-category compatibility declarations.
  • Normative: the OWL axioms and annotations in the vocabulary document and the constraints in the core SHACL profile. Explanatory: this page, the README, examples, and the recommended/diagnostic profiles.
  • PAX deliberately has no upper-ontology grounding: for document structure the SPAR suite is the community layer, and reuse of living vocabularies takes precedence — a considered decision, not an omission.

SHACL profiles — severity contract

ProfileSeverityMeaning
coreViolation Structural conformance. Conformance to PAX = zero core violations.
recommendedWarning Interoperability + materialized-projection consistency (pax:matter/pageStart/pageEnd must agree with typing and containment).
diagnosticInfo Data-quality signals (e.g. text-less paragraphs) — surfaced, never blocking.

Exact validation command (fast layer, includes all three profiles + term-coverage checks):

uv run --with rdflib --with pyshacl python validate.py

Scoring concerns — IoU between regions, coordinate tolerances, reading-order distance — are evaluation metrics computed by a separate engine. SHACL severity has no tolerance semantics; the profiles check conformance only.

Imports and dependency policy

  • PAX declares no owl:imports. External terms (DoCO, DEO, FaBiO, PO, OA, PROV, SKOS, FRBR, DCTerms, FOAF) are reused term-level, with local category declarations to keep OWLAPI tooling from guessing (anti-punning).
  • The pinned dependency closure is a reproducibility artifact for the formal checks (HermiT consistency, profile gate) — it is not the normative graph of those ontologies. Regenerate deliberately with fetch_imports.py to bump the pins.
  • Slow layer (OWL 2 DL gate on the standalone document, HermiT over the transformed closure, ROBOT lint, JSON-LD round-trip): uv run --with rdflib --with pyld python formal_checks.py (needs Java).

Versioning

Term IRIs are stable and unversioned; releases are snapshotted under releases/<version>/ and referenced by owl:versionIRI (current: …/releases/0.2.0/vocab). Deprecation uses owl:deprecated — terms are never deleted or reassigned. Full policy: VERSIONING.md.

Using PAX

  • Worked example: Mercadier 2011 — a complete 78-node export of a real 6-page paper (produced by the deepseek-ocr-experiments exporter; instance IRIs minted under the producing application's namespace, default https://w3id.org/alexandria/paper/…).
  • JSON-LD context: context.jsonld — all prefixes + term shortcuts.
  • Competency questions the graph is designed to answer (with SPARQL): COMPETENCY_QUESTIONS.md.
  • pax:chartData contract: an rdf:JSON literal holding extracted chart payloads as immutable evidence. The OWL vocabulary deliberately carries no range axiom (rdf:JSON is outside the OWL 2 datatype map); the datatype is enforced at the instance level by paxs:ChartDataShape, which is negative-tested in CI.
  • Materialized conveniences: pax:level, pax:matter, pax:pageStart/End are classifier outputs (evaluation targets) materialized as flat properties; the recommended profile checks they agree with the containment structure.
